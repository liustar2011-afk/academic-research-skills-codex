# 研究项目工作机制

本目录定义整个工作区通用的研究项目机制。它不属于某一个具体研究项目，而是所有研究项目共用的操作系统。

## 目标

解决长期研究中的四类问题：

1. 中断后快速恢复。
2. 不同研究项目相互隔离。
3. 不同阶段成果可留存、可版本化、可回溯。
4. 主线任务与支线深度研究可以分工，并把支线成果有序汇入主线。

## 核心概念

| 概念 | 含义 |
|---|---|
| 项目 | 一个独立研究主题，拥有自己的材料、提纲、模型、草稿和管理文件 |
| 主线 | 项目的核心研究问题、理论框架、论文结构和最终成果 |
| 支线 | 为主线中的某个局部问题单独展开的深度研究 |
| 阶段成果 | 某一阶段形成的可复用成果物，如材料诊断、开题框架、文献综述 |
| 依赖关系 | 上游成果物修改后会影响的下游成果物 |
| 汇入 | 将支线研究成果经审查后吸收到主线文件中 |

## 推荐目录

所有研究项目放入：

```text
research-projects/
  项目名称/
```

每个项目建议遵循 `templates/project-template/` 的结构。

## 引入的研究能力

已从外部技能库引入并转化为本工作区可复用能力：

| 能力模块 | 入口 | 用途 |
|---|---|---|
| 证据系统 | `evidence/` | 管理事实卡、来源清单、证据池、证据缺口和待核验事项 |
| 研究方法 | `methods/` | 支持咨询式研究框架、假设驱动研究和问题拆解 |
| 领域适配器 | `domain-adapters/` | 为国家数据局、国家能源局、电力数据基础设施、中电联等材料建立统一口径 |
| 交付转化 | `delivery/` | 将研究成果转为 Word、PPT、公文等交付件 |
| 质量闸门 | `quality-gates/` | 在阶段推进、支线汇入、成果定稿前执行检查 |
| 原始技能归档 | `imported-skills/` | 保留外部 14 个技能原文，便于追溯 |

## 通用工作流

1. 创建项目目录。
2. 固化项目说明、研究问题和研究边界。
3. 建立材料索引、来源清单、证据池和证据缺口。
4. 选择领域适配器，明确本项目使用的政策和行业口径。
5. 产出提纲、理论框架、模型和草稿入口。
6. 建立支线研究目录。
7. 对支线研究使用标准任务单。
8. 支线完成后先形成成果包，再汇入主线。
9. 主线更新后记录变更，并检查依赖影响。
10. 需要交付时，再按 Word、PPT 或公文规范生成输出件。

### Mermaid 可视化

研究项目 Markdown 可以直接嵌入 Mermaid 图，用来表达任务流、成果依赖和支线汇入路径。优先在以下文件中使用：

- `PROJECT_INDEX.md`：展示当前任务流和阶段推进。
- `DEPENDENCY_MAP.md`：展示上游成果物到下游成果物的影响关系。
- `07_branch_research/BRANCH_INDEX.md`：展示多条支线如何汇入主线。

任务流示例：

```mermaid
flowchart LR
  A[项目说明] --> B[研究问题]
  B --> C[证据池]
  C --> D[理论框架]
  D --> E[论文提纲]
  E --> F[草稿与交付件]
```

## 阶段体系

所有项目使用统一阶段定义：

- `PHASES.md`

项目状态应引用 `P0` 至 `P6` 的阶段名，不使用未定义的临时阶段描述。

## 关键文件

每个项目根目录应至少包含：

| 文件 | 用途 |
|---|---|
| `PROJECT_INDEX.md` | 项目控制台，最快入口 |
| `CURRENT_STATE.md` | 中断恢复状态 |
| `CHANGELOG.md` | 版本和变更记录 |
| `DEPENDENCY_MAP.md` | 依赖关系和影响分析 |
| `REVISION_QUEUE.md` | 待修订事项队列 |
| `DECISIONS.md` | 关键研究决策 |
| `研究边界.md` | 研究范围和排除项 |
| `HYPOTHESIS_LOG.md` | 研究假设和验证状态 |
| `09_evidence/` | 证据系统 |
| `10_domain_adapter/` | 领域适配器选择 |
| `06_synthesis/research_asset.md` | 研究成果资产母本 |
| `review_log.md` | 复核记录 |
| `artifact_manifest.md` | 成果清单 |

## 支线研究原则

支线研究不得直接改主线核心文件，除非用户明确要求。默认流程是：

```mermaid
flowchart LR
  A[支线任务单] --> B[支线深度研究]
  B --> C[支线成果包]
  C --> D[汇入建议]
  D --> E[主线影响分析]
  E --> F[主线文件更新]
  F --> G[变更记录]
```

## 文件状态约定

| 状态 | 含义 |
|---|---|
| draft | 初稿，可大幅修改 |
| baseline | 当前阶段基准稿 |
| review | 待检查或待确认 |
| validated | 已确认，可作为下游依据 |
| integrated | 已汇入主线 |
| superseded | 已被新版本替代 |
| final | 定稿 |

## 使用建议

新建项目时，复制 `templates/project-template/`。

新建支线时，复制 `templates/branch-template/`。

支线汇入主线前，使用 `workflows/branch-integration-checklist.md` 检查。

## 质量闸门

建议在以下节点执行质量闸门：

| 节点 | 检查重点 |
|---|---|
| 项目启动 | 研究边界、研究问题、资料来源是否明确 |
| 阶段成果形成 | 结论是否可追溯到材料或证据 |
| 支线汇入 | 支线输出是否说明服务的主线任务、汇入位置和影响范围 |
| 上游成果修改 | 是否检查并更新下游提纲、模型、草稿和任务板 |
| 交付输出 | Word、PPT 或公文是否与当前研究成果资产一致 |
