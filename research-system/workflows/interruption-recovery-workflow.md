# 中断恢复工作流

## 一、恢复入口

每次恢复项目，先读：

1. `CURRENT_STATE.md`
2. `PROJECT_INDEX.md`
3. `CHANGELOG.md`
4. 本次任务相关文件

## 二、恢复时要判断

1. 当前项目处在哪个阶段？
2. 当前基准成果是什么？
3. 最近一次修改了什么？
4. 有没有未完成的修订队列？
5. 有没有正在进行的支线研究？
6. 本次任务是继续主线、修改上游，还是启动支线？

## 三、恢复后输出

恢复后先给出简短状态：

```text
当前项目处于：……
最近完成：……
下一步建议：……
本次如果继续，我建议先处理：……
```

## 四、往回修改时

如果用户要修改上游成果，先读：

1. `DEPENDENCY_MAP.md`
2. `REVISION_QUEUE.md`
3. `DECISIONS.md`

然后输出影响分析：

```text
修改对象：……
预计影响：……
需要同步检查的文件：……
建议处理顺序：……
```

## 五、恢复禁忌

1. 不凭聊天记忆直接继续。
2. 不跳过 `CURRENT_STATE.md`。
3. 不直接覆盖基准稿。
4. 不在未做影响分析时大改研究问题、理论框架或论文提纲。
